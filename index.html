timberdbGostorage engine
A time-partitioned, TTL-native LSM storage engine with higher append throughput than Pebble and Badger in benchmarks.
▼
The problem
Time-series and log workloads are append-heavy and time-ordered, but general-purpose key-value stores pay for features these workloads never use — point lookups, cross-level compaction, tombstone churn — which drags down write throughput and storage efficiency.
The stack & why
  • Go — fine control over allocations and a clean concurrency model for the write, compaction, and sweeper paths.
  • LSM tree (WAL → memtable → SSTable) — append-optimized writes with sequential I/O.
  • Time partitioning — range scans skip whole partitions by timestamp instead of scanning everything.
  • zstd at flush time — compression runs off the hot write path, so it costs storage savings without write latency.
  • Native TTL sweeper — expired data is dropped by removing whole SSTables, avoiding tombstone write-amplification.
Key technical challenge
Exceeding mature engines' append throughput while keeping full fsync durability. Partition-local sequential writes need exactly one fsync per record with no cross-level compaction amplification; zstd runs only at memtable-flush time; and the WAL encoder reuses a single pre-allocated buffer per instance, cutting the hot path to 3 allocations per write to minimize GC pressure.
Impact — measured benchmarks
2.0× Pebble append5.8× Badger13× bbolt3 allocs/opscan 6.6× Badger

svid-exchangeGozero-trust auth
A Zero Trust service that trades a SPIFFE SVID for a scoped, short-lived JWT — no static secrets.
▼
The problem
Microservices commonly authenticate to each other with long-lived static shared secrets. They’re painful to rotate, and a breach of any single secret opens the door to lateral movement across the whole system.
The stack & why
  • Go + gRPC — a typed, protobuf-defined API surface for the exchange.
  • SPIFFE / SPIRE — standard workload identity, so trust is tied to the workload, not its network location.
  • mTLS — mutually authenticates the caller before a single token is issued.
  • ES256 JWT — asymmetric, short-lived, verifiable by any service without sharing a secret.
  • Docker Compose dev stack — one command brings up SPIRE Server + Agent + the service with workloads pre-registered.
Key technical challenge
Issuing a least-privilege, short-lived token to a caller proven only by its SVID over mTLS — with no shared secret anywhere in the path. The full SPIRE-backed dev and end-to-end test setup verifies the trust chain from workload attestation through to a scoped JWT the target service can validate offline.

saga-conductorGodistributed txns
A saga orchestrator for distributed transactions — single binary, gRPC, no database.
▼
The problem
A transaction that spans several services can’t use one database transaction. When a step halfway through fails, the system is left in a partially-applied, inconsistent state with no automatic way back.
The stack & why
  • Go — compiles to a single static binary that’s trivial to deploy.
  • gRPC + protobuf — a typed orchestration API for defining and running sagas.
  • Saga pattern — compensating actions instead of distributed locks, so no external database is needed to coordinate state.
Key technical challenge
Guaranteeing that on any step failure, every completed step is rolled back in exact reverse order via its compensating action — and doing it as a single self-contained binary with no backing datastore. Callers just declare each step’s forward and compensating action; the conductor owns ordering, execution, and recovery.
