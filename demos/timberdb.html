timberdb interactive

A time-partitioned, TTL-native LSM storage engine for append-heavy, time-ordered data (logs, metrics, events). Writes flow WAL → memtable → SSTable; data is bucketed into time windows so range scans skip whole partitions by timestamp; and expired data is reclaimed by dropping entire SSTables — no tombstones, no cross-level rewrite. Benchmarks: 2.0× Pebble / 5.8× Badger / 13× bbolt append throughput at 3 allocs/op.

The engine is real: the sorted-slice memtable (binary-search insertion by (timestamp, sourceID)), the time-window routing (floorDiv(ts, window)), the FIFO compaction trigger + k-way merge, the range-scan partition/footer skip, and the TTL sweep all run the same algorithm as the Go source. Only disk I/O is simulated — SSTables live as in-memory objects so you can watch them.

The life of a record

Follow one batch of sensor readings from a durable write all the way to expiry: appended to the WAL, routed to its hour-window partition, sorted in the memtable, flushed to an immutable SSTable, FIFO-compacted, range-scanned (skipping a partition that can't overlap), and finally dropped wholesale by the TTL sweeper.     02 / sandbox

    Drive the engine yourself

    Append records and watch them route into time-window partitions and fill each partition's sorted memtable. When a memtable hits its flush threshold it freezes into an immutable SSTable; when a partition exceeds its file limit, FIFO compaction merges them. Then run a range scan to see which partitions and SSTables are read versus skipped, and a TTL sweep to drop expired SSTables whole.

    engine config — internal/engine/options.go (values shrunk so flush & compaction are watchable)
    Real defaults: MemtableSizeBytes = 64 MiB, MaxFilesPerPartition = 10, PartitionDuration = 1 h. Changing config rebuilds the store.
    append a record — Engine.Append(rec)
    range scan — Engine.Scan(start, end) · footer + partition skip
    TTL retention sweep — Engine.sweepRetention() · drop SSTables where maxTs < now − retention

    Everything above runs entirely client-side. The storage engine is faithful to the Go implementation: the sorted-slice memtable's binary-search insertion ordered by (Timestamp, SourceID) (internal/memtable/memtable.go), time-window routing via floorDiv(ts, window)·window (internal/partition/router.go), the flush trigger at the memtable size threshold, FIFO compaction merging all of a partition's files once they exceed the limit via a k-way min-heap (internal/compaction/fifo.go, merge.go), range scans that skip partitions whose window can't overlap and SSTables whose footer min/max can't overlap (internal/sstable/reader.go), and the TTL sweep dropping whole SSTables where MaxTimestamp < now − RetentionDuration (engine.go · sweepRetention). Only disk I/O is simulated — SSTables are in-memory objects and the config values are shrunk so flush/compaction happen on a handful of records instead of 64 MiB.
