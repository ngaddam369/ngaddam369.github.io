saga-conductor interactive

A saga orchestrator for distributed transactions. A saga is a sequence of HTTP steps across services; when any step fails, the conductor automatically rolls back the completed steps in reverse order via their compensating actions — no distributed locks, no external database.

The engine is real: the saga & step state machines, the reverse compensation, and the retry backoff (rand(0, base·2^attempt)) run the same algorithm as the Go source. The downstream services are simulated so you can choose how each one responds.

    02 / sandbox

    Run your own saga

    Pick a saga, decide how each step's forward and compensating call behaves, then start it. The real engine drives the steps forward, retries failures with exponential backoff, and compensates in reverse on failure. Then try the edge cases: dead-letter a compensation, crash mid-run and resume, abort, or replay an idempotency key.

    step name forward call compensate retries* backoff ms* timeout s
    *Per the engine, retries/backoff of 0 fall back to the server defaults (3 retries, 100 ms). Total attempts = retries + 1. flaky fails its first 2 attempts then succeeds — so it only recovers if retries allow a 3rd attempt.
    Reuse a key to see CreateSaga return the existing saga instead of running a duplicate.

    Everything above runs entirely client-side. The orchestration engine is faithful to the Go implementation: the saga & step state machines (with the same valid-transition checks from internal/saga/statemachine.go), the forward → reverse-compensation algorithm (skip non-succeeded steps; halt and dead-letter on the first compensation failure), the retry loop with exponential backoff and full jitter (rand(0, base·2^attempt)), idempotency-key dedup, abort, and crash recovery all mirror internal/engine/engine.go. The downstream HTTP services are simulated (you choose each response) and animation timings are compressed so a run is watchable — the backoff values shown are computed with the real formula.
